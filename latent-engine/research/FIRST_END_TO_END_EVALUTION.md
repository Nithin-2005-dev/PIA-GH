# Major Observations

## 1. End-to-End Architecture Works

Every subsystem communicated correctly.

No manual data conversion was required after commit collection.

The architecture proved that:

Repository → Intelligence is operational.

---

## 2. Large Knowledge Expansion

Only twenty commits generated

173 Evidence

145 Modules

12 Developers

This indicates that one commit contributes knowledge to many files rather than only one.

---

## 3. Repository Coverage is Extremely Sparse

Although 145 modules were observed,

only twenty commits were analysed.

Therefore most modules contain incomplete knowledge.

Current ownership should be interpreted as

Observed Ownership

rather than

Actual Ownership.

---

## 4. Snapshot Intelligence

The current engine analyses only the latest observed commits.

Historical repository evolution is ignored.

Therefore the current graph represents

Current Observation

instead of

Knowledge Evolution.

---

## 5. Commit Collection is the Bottleneck

Evidence extraction completed almost instantly.

GitHub API collection required the majority of execution time.

Current bottleneck

GitHub REST requests.

---

## 6. Evidence Pipeline is Efficient

Commit collection

≈ tens of seconds

Evidence extraction

≈ milliseconds

Projection building

≈ milliseconds

The intelligence engine itself is computationally inexpensive.

---

## 7. Expertise Appears Reasonable

Developers naturally accumulated expertise based on observed commits.

However the model currently assumes

More modifications

⇒

More expertise

which is only partially true.
